# DSC (Decentralized Stablecoin) — 설계 노트

**목표:** 초과담보 기반, 알고리즘 페그(algo peg) 없는 탈중앙 스테이블코인. USD 페그. MakerDAO/DAI 계열, DSC 강의 구조 참고.

**핵심 불변식(invariant):** 언제나 `담보의 USD 가치 > 발행된 DSC의 USD 가치`. 아래 모든 결정은 이 불변식을 지키기 위한 것.

---

## 1. 담보 자산

**선택:** WETH, WBTC. 테스트는 mock ERC-20 + mock price feed로 가격을 결정론적으로 조작.

**왜:**
- **exogenous(외생) 담보만** 사용. 프로토콜 자체 토큰을 담보로 쓰지 않음 → UST/LUNA식 death spiral 원천 차단.
- WETH/WBTC는 온체인 유동성·오라클 커버리지가 가장 두껍고 검증됨. 청산 시 담보를 시장에서 처분하기 쉬움(유동성이 청산 시스템의 전제).
- 둘 다 crypto-native ERC-20이라 통합이 단순.

**대안 / 왜 안 골랐나:**
- **단일 담보(WETH only):** 구현은 더 단순하지만 자산 집중 리스크. 다중 담보로 최소한의 분산 확보.
- **알트코인·LP 토큰:** 담보 다양성·수익은 ↑지만 유동성이 얇고 변동성이 커서 "담보를 못 파는" 청산 실패 위험.
- **스테이블코인(USDC 등)을 담보로:** 변동성은 낮지만 "스테이블코인 담보로 스테이블코인 발행"은 목적이 모호하고 depeg/동결 리스크를 상속.
- **endogenous(자체 토큰) 담보:** 자본효율은 좋아 보이지만 담보=부채가 같은 가격에 묶여 시장 붕괴 시 동반 폭락 → 구조적으로 금지.

**트레이드오프:** WETH·WBTC는 검증됐지만 둘 다 "crypto beta"라 상관관계가 높음. 시장 전체 급락 시 동시에 무너져 분산 효과는 제한적. 진짜 비상관 분산엔 RWA 같은 담보가 필요하지만 범위 밖.

**면접 한 줄:** *"왜 자체 토큰을 담보로 안 씀?"* → 자체(endogenous) 담보는 담보와 부채가 같은 리스크에 노출돼 death spiral을 만든다(UST 사례). 외생 담보만 받아 그 커플링을 끊었다.

## 2. 담보비율 · 청산 임계값

**선택:** 초과담보 **200%** (= `LIQUIDATION_THRESHOLD = 50%`). 부채 $1당 담보 최소 $2. HF는 담보의 50%까지만 부채로 인정.

**왜 이 숫자:**
- ETH/BTC 일간 변동성이 큼(하루 -20~30%, 나쁜 주엔 -50%+ 가능). 청산이 오라클 하트비트·가스 경쟁 때문에 즉각적이지 않으므로, 청산 실행 전까지 `담보 > 부채`가 유지될 버퍼가 필요.
- 200%면 담보가 반토막(-50%) 나도 원금 커버 라인. 청산 보너스(10%)까지 지불할 여유를 담보 안에 내장.
- Maker ETH-A의 150%보다 보수적. 학습/데모 시스템이라 안전 마진을 우선.

**대안 / 왜 안 골랐나:**
- **150% (Maker ETH-A 급):** 자본효율은 ↑(같은 담보로 더 많이 발행)지만 버퍼가 얇아 변동성 구간에서 청산·부실 위험 ↑.
- **110~120% (저변동 담보용):** 매우 효율적이지만 WETH/WBTC 변동성엔 부적합. 순식간에 under-collateralized.
- **300%+:** 안전하지만 자본 비효율 → 아무도 안 씀(발행 유인 소멸).

**트레이드오프:** 자본효율 vs 지급능력. 낮은 비율 = 차입력↑·유저 매력↑지만 청산/부실 리스크↑. 200%는 데모 안전성에 기운 선택.

**면접 한 줄:** *"왜 하필 200%?"* → 담보의 변동성(반토막 시나리오)과 청산에 필요한 보너스·지연을 모두 흡수하려면 최소 2배가 필요. Maker의 150%보다 보수적으로 잡아 "부실 없음"을 우선했다.

## 3. Health Factor 정의

```
adjustedCollateral = collateralValueUsd * LIQUIDATION_THRESHOLD / 100   // × 50%
healthFactor       = adjustedCollateral * 1e18 / totalDscMinted
```

- `HF ≥ 1e18` (= 1.0): 안전
- `HF < 1e18`: 청산 가능
- `totalDscMinted == 0`: 부채 없음 → HF = ∞ (max)

**의미:** "담보를 청산임계값으로 할인한 가치가 부채의 몇 배인가." 1.0이면 딱 경계.

**예시:**

| 담보 | 부채 | 계산 | HF | 상태 |
|---|---|---|---|---|
| $200 | $100 | (200×0.5)/100 | 1.0 | 경계 |
| $180 | $100 | (180×0.5)/100 | 0.9 | 청산 대상 |
| $200 | $50  | (200×0.5)/50  | 2.0 | 안전 |

**구현 노트:** 정수 연산이라 `PRECISION(1e18)` 스케일링 필요. **먼저 곱하고 나중에 나눠서** 정밀도 손실 최소화.

**면접 한 줄:** *"HF가 뭘 측정하나?"* → 청산임계값으로 할인한 담보가치 ÷ 부채. 1 밑으로 떨어지면 청산 트리거. Aave와 같은 개념이고, 담보비율을 유저 관점의 단일 지표로 표현한 것.

## 4. 청산 인센티브 (Liquidation Bonus)

**선택:** **10% 보너스** (`LIQUIDATION_BONUS = 10`). 청산자가 부채 $100 상환 시 담보 $110어치 수령.

**왜 인센티브를 주나:**
- 청산은 익명의 제3자(청산봇)가 **자발적으로** 수행. 그들은 가스비 + DSC 조달 + 가격 리스크를 감수 → 순이익이 없으면 아무도 안 함.
- 청산이 안 되면 프로토콜에 bad debt 누적 → 페그·지급능력 붕괴. 즉 보너스는 **"지급능력을 지키는 비용"**.
- 10%면 대부분의 가스·MEV 경쟁 환경에서도 청산봇이 달려들 만한 마진.

**대안 / 왜 안 골랐나:**
- **0% (보너스 없음):** 청산 유인 0 → 부실채권 방치. 실패.
- **5%:** 자본은 절약되지만 고가스·고변동 구간에서 유인 부족, 청산 지연 위험.
- **15~20%:** 청산은 빨라지지만 (a) 청산당한 유저 손실 과대, (b) 버퍼를 더 빨리 소모 → 더 높은 담보비율을 강제해야 함.
- **경매 방식(Maker collateral auction):** 가격 발견이 정교하고 자본효율 좋지만 구현 복잡도 급증(입찰·keeper 등). 범위 밖.

**트레이드오프:** 유인 강도 vs 유저 보호 vs 버퍼 소모. 보너스는 초과담보 버퍼에서 지급되므로 §2의 담보비율과 **직접 연동**. 200% 버퍼가 10% 보너스를 감당하도록 설계.

**면접 한 줄:** *"왜 청산자에게 보너스를?"* → 청산은 강제할 수 없는 자발적 행위라 이익이 있어야 실행된다. 10%는 청산봇이 가스·MEV를 감수하고도 남는 마진이자, 초과담보 버퍼가 흡수할 수 있는 수준.

## 5. 오라클: Chainlink Price Feed

**선택:** Chainlink `AggregatorV3` price feed. + 스테일 방어 래퍼(`OracleLib`: `updatedAt` 확인 후 heartbeat 초과 시 revert).

**왜 (하드코딩 대비):**
- HF·청산이 전부 실시간 가격에 의존. 하드코딩 = 가격이 안 움직임 = 청산 로직 자체가 무의미(현실과 즉시 괴리).
- Chainlink는 다수 노드 집계 + 온체인 검증 + 광범위 채택 → 단일 소스 조작에 강함. WETH/WBTC 커버리지 확실.
- 스테일 체크를 붙여 **"오라클이 멈추면 시스템도 멈춘다(fail-safe)"** — 잘못된 가격으로 잘못 청산하는 것보다 안전.

**대안 / 왜 안 골랐나:**
- **하드코딩 가격:** 구현 비용 0이지만 비기능적. 특정 단위테스트 외 무용.
- **DEX 스팟(Uniswap spot):** 온체인이지만 flash loan으로 1블록 내 조작 가능 → 청산 남용.
- **Uniswap TWAP:** 스팟보다 조작 저항은 ↑지만 단일 소스·지연이 있고, 자본이 있으면 여러 블록에 걸쳐 조작 가능.
- **자체/중앙 가격 서버:** 단일 실패점·신뢰 가정·검열 리스크.
- **Pyth 등 pull 오라클:** 유효한 대안(저지연). 다만 Chainlink push 모델이 통합이 단순하고 강의 자료와 일치해 우선.

**트레이드오프:** Chainlink는 신뢰성이 높지만 외부 의존성 추가 → 스테일·하트비트·피드 중단을 반드시 핸들링해야 함(`OracleLib`). 탈중앙성·안전 ↔ 지연·의존성.

**면접 한 줄:** *"하드코딩 대신 Chainlink 쓴 이유?"* → 청산이 실시간 가격에 걸려 있어 정확·조작저항·최신 가격이 필수. 단일 DEX는 flash loan 조작, 하드코딩은 비기능. 스테일 체크까지 붙여 오라클 실패 시 청산 대신 "정지"하도록 했다.

## 6. Known Limitation — 극단적 폭락 시 청산 실패 가능성 (Future Work)

**문제:** 초과담보(200%)와 청산 보너스(10%)는 "평상시 변동성"을 가정한 완화책이지, 극단적 폭락에서 청산이 100% 보장된다는 뜻이 아니다. 실제로 2020년 3월 MakerDAO "Black Thursday" 때 이 가정이 깨졌던 전례가 있다(가스비 폭등으로 청산 미실행, 오라클 지연, 경매가 0원에 낙찰되는 사례 등으로 대규모 bad debt 발생).

**깨질 수 있는 지점:**
- **가스비 폭등:** 패닉 시 네트워크 혼잡 → 청산 실행 비용이 보너스(10%)를 초과 → 청산봇이 안 움직임.
- **오라클 지연(staleness):** 가격이 초 단위로 급락하는데 오라클은 heartbeat 주기로만 갱신 → 담보가 이미 부채 밑인데 컨트랙트는 옛 가격으로 "안전"이라 착각.
- **담보 매도 슬리피지:** 청산자가 받은 담보를 시장에서 팔아야 이득 실현인데, 동시다발 매도로 슬리피지 커지면 청산 매력 급감.
- **속도 자체의 한계:** 가격 낙폭이 버퍼(200%)보다 빠르면, 청산 트랜잭션이 도착하기 전에 이미 담보 < 부채(underwater) 상태가 되어 청산자도 손해라 안 함.

**현재 범위 내 대응:** 오라클 스테일 체크(§5)만 반영. 나머지는 범위 밖.

**나중에 깊게 다룰 방향(To revisit):**
- Circuit breaker / 긴급 정지 메커니즘
- Dutch auction식 부분 청산 (Maker Liquidations 2.0 참고)
- 오라클 heartbeat를 더 촘촘하게, 또는 다중 오라클 소스 교차검증
- 담보비율을 시장 변동성에 따라 동적으로 조정(정적 200% 대신)
- bad debt 발생 시 시스템 차원의 backstop(보험 기금, 거버넌스 토큰 발행 등) 설계

**면접 한 줄:** *"이 시스템의 한계는?"* → 초과담보는 "평시" 리스크를 흡수하는 설계지, 극단적 시나리오(가스 폭등+오라클 지연+슬리피지가 동시에 겹치는 경우)에서 청산 실패와 bad debt은 구조적으로 남아있다. Black Thursday가 실증 사례이며, 이는 이후 심화 학습 대상으로 남겨둠.

## 부록: 상수 요약

| 상수 | 값 | 의미 |
|---|---|---|
| `LIQUIDATION_THRESHOLD` | 50 | 담보의 50%만 부채로 인정 (= 200% 초과담보) |
| `LIQUIDATION_BONUS` | 10 | 청산자 10% 보너스 |
| `LIQUIDATION_PRECISION` | 100 | 위 백분율의 분모 |
| `MIN_HEALTH_FACTOR` | 1e18 | HF 하한 (= 1.0) |
| `PRECISION` | 1e18 | 고정소수 스케일 |

> 숫자 근거는 MakerDAO 파라미터 / DSC 강의에서 차용, "왜"는 본 문서의 판단.
